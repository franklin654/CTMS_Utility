# Phase 5 — Visit Management: Completion Report

**Status:** Complete
**Scope:** Backlog Epic 4 (Stories 01–04): configure visit schedules, track completed & missed visits, generate visit alerts & notifications, view visit schedule & history. Extended beyond the literal BRD with two confirmed additions: template-backfill onto already-enrolled subjects, and ad-hoc (unscheduled) visits.

---

## 1. What Was Built

### Backend (`com.ctms.ctms_backend.visit`)

- **Schema** (`V7__visit_management.sql`, `V8__adhoc_visits.sql`): `visit_template` (per-study, per-sequence config: target day, early/late window, procedures, onsite/remote, soft-deactivate) and `visit` (subject-specific instance, denormalized snapshot of the template fields at generation/last-sync time, `scheduled_date`, status, actual date/time/notes, `reason_code`, self-referencing `rescheduled_from_visit_id` for the reschedule trail). `V8` makes `visit_template_id` nullable to support ad-hoc visits.
- **Visit generation is config-driven, not hardcoded**: `VisitSchedulingService.generateForSubject` fires from `SubjectService.enrollSubject` right after a successful enrollment, creating one `Visit` per active template with `scheduledDate = subject.screeningDate + template.targetDay` — this was a confirmed design choice since the BRD names "target day" and "screening date" as fields without ever stating the anchor relationship between them.
- **Template edits propagate live**: `VisitTemplateService.update` re-syncs any of that template's still-`SCHEDULED` visits (denormalized fields + recomputed `scheduledDate`) without touching `COMPLETED`/`MISSED`/`RESCHEDULED` rows — satisfies AC4 ("updated visit definitions propagate without losing historical visit data").
- **State machine**: `SCHEDULED → COMPLETED` / `SCHEDULED → MISSED` (mandatory reason code) / `SCHEDULED → RESCHEDULED` (mandatory reason code, creates a new linked `SCHEDULED` row via `rescheduledFromVisit` rather than mutating the date in place) — all guarded in `VisitService`, mirroring the Study/Subject lifecycle pattern (dedicated action per terminal transition, not a raw enum setter).
- **Compliance rate** (BRD gives no formula): `completed / (completed + missed)` among a subject's due visits (`scheduledDate <= today`); not-yet-due `SCHEDULED` visits and ad-hoc visits are excluded from the denominator entirely.
- **Visit alerts** (`VisitAlertService`): the first `@Scheduled` job in this codebase (daily 6am cron, `common/config/SchedulingConfig`), sweeping for "due tomorrow" and "overdue by X days" (literal BRD examples), deduped via a new `NotificationRepository.findByRecipientAndTypeAndLink` check so the same alert never re-sends. "Visit missed" fires synchronously from `VisitService.markMissed` instead of the batch sweep, per the BRD's third example being a direct consequence of the status change rather than a scheduled check.
- **Two extensions beyond the literal BRD**, both explicitly discussed and confirmed with you before implementation:
  1. **Template backfill**: adding a *new* `VisitTemplate` to a study that already has enrolled subjects now generates the missing visit for every subject still active in the trial (`SCREENED`/`ENROLLED`/`IN_TREATMENT`) — `COMPLETED`/`WITHDRAWN` subjects are skipped since their participation is formally closed. The BRD only describes propagation on template *edits*, not on templates added after enrollment; this closes that gap.
  2. **Ad-hoc (unscheduled) visits**: `POST /api/subjects/{id}/visits/ad-hoc` lets a Site Coordinator, Investigator, Study Manager, or Admin schedule a one-off visit outside the protocol schedule (AE follow-up, dose-modification check) with a mandatory reason. `Investigator` is intentionally scoped into this one endpoint only — they remain unable to mark protocol visits completed/missed/rescheduled — since ad-hoc visits are typically clinically motivated. Ad-hoc visits are excluded from compliance rate.
- **Visit ordering switched to chronological** (`findBySubjectIdOrderByScheduledDateAsc`, was by `sequenceNumber`) — this more literally satisfies Story 04 AC1 ("full visit history displayed in chronological order") and lets ad-hoc visits interleave correctly with protocol visits in the schedule view.
- **RBAC**: `VisitTemplateController` — write `STUDY_MANAGER`/`ADMIN`, read also includes `SITE_COORDINATOR` (mirrors the Phase 4 eligibility-criteria precedent — Site Coordinators need to see templates even though they don't define them). `VisitController` — write (complete/miss/reschedule) `SITE_COORDINATOR`/`STUDY_MANAGER`/`ADMIN`; ad-hoc-visit write additionally includes `INVESTIGATOR`; read is all internal roles except `PATIENT_SUBJECT`.
- **Audit + notifications**: every template create/update/deactivate and every visit create/state-change writes to `AuditLog`. Lifecycle-relevant notifications (`VISIT_MISSED`, `VISIT_DUE_TOMORROW`, `VISIT_OVERDUE`) go to the subject's creator and the site's assigned CRA, reusing Phase 3/4's `Site.assignedCra` pattern.

### Frontend (`features/visit-templates/`, `features/subjects/subject-detail/`)

- `visit-templates` — a standalone admin page (linked from `study-detail` next to "Eligibility Criteria") for `STUDY_MANAGER`/`ADMIN` to create/edit/deactivate templates.
- `subject-detail` (extended) — a new "Visit Schedule" card: compliance-rate badge, chronological visit list with `MISSED`/`RESCHEDULED` rows highlighted amber, inline Mark-Completed/Mark-Missed/Reschedule forms per `SCHEDULED` row, and a "Schedule Ad-Hoc Visit" form (gated to `SITE_COORDINATOR`/`INVESTIGATOR`/`STUDY_MANAGER`/`ADMIN`) with ad-hoc visits tagged with a purple "Ad-hoc" badge instead of a sequence number.

## 2. Defects Found & Fixed

1. **`VisitTemplateService` never set `createdBy`/`modifiedBy`** (design-time, caught while writing the integration test) — the migration requires these `NOT NULL`, so the very first template creation would have thrown a DB constraint violation. Fixed by threading the actor's username through `create`/`update`/`deactivate`, matching `StudyService`'s pattern.
2. **`VisitAlertService.runDailyAlertSweep` was `@Transactional(readOnly = true)`** (runtime, caught during manual verification against the real dev DB by temporarily shortening the cron and watching it actually fire) — the sweep triggers notification inserts via `NotificationService.notify`, which inherited the read-only transaction and failed with "cannot execute INSERT in a read-only transaction." Fixed by dropping `readOnly = true`; re-verified live that both `VISIT_DUE_TOMORROW` and `VISIT_OVERDUE` alerts fire correctly and dedupe on a second sweep.
3. **`mat-label` + `placeholder` overlap on the ad-hoc visit form** (runtime, caught during the browser walkthrough) — Angular Material's floating label doesn't reliably float when a field has both a label and a placeholder and is empty/unfocused, so the two rendered on top of each other. Fixed by moving the example text into `mat-hint` (helper text below the field) instead of `placeholder`, which is the Material-supported way to show that guidance without the conflict.

None of these surfaced as customer-visible issues in earlier phases' patterns re-applied here (e.g. dedicated exceptions instead of raw `IllegalArgumentException`, the `:search = ''` null-binding fix) — this phase's new defect classes were specific to the new scheduled-job and actor-threading code paths introduced here.

## 4. Known Gaps / Carried-Forward Items

- **Compliance-rate aggregation is subject-level only.** No study/site-level rollup exists yet — the BRD gives no formula or aggregation level at all, and cross-study/site dashboards are Epic 6 territory, out of scope here.
- **Visit-dependency rules** ("Visit 2 cannot occur before Visit 1", Epic 9 Story 02 AC3) are not implemented — `RuleSet.CATEGORY_VISIT` remains seeded but unused, reserved for the Phase 9/System Configuration no-code rules work.
- **Alert thresholds are fixed, not per-study configurable** — "due tomorrow" and "overdue" use the BRD's literal wording as fixed day-offsets rather than a per-study rule, since the BRD gives no indication these should vary by study.
- **No dedicated `VisitStatusHistory` table** — a Visit only ever makes one terminal transition, so `AuditLog` plus the `rescheduledFromVisitId` chain is the full lineage; this was a deliberate simplification versus Study/Site/Subject's separate history tables.
- Same Docker/Testcontainers gap as prior phases — integration tests use the dedicated `ctms_testdb` Postgres database instead. Also carried forward from this session specifically: the `ctms` Postgres role/table-ownership setup needed one-time repair in this environment (role didn't exist, then didn't own its own tables) — documented here in case it recurs in a fresh environment.
